    2. PHP7 相对于PHP5， 有哪些提升？

    这个当初回答并不太好， 现在贴出下面的整理之后的内容

    • 引入了严格模式开关

    • 改进了错误处理机制
      很多致命错误以及可恢复的致命错误都被转换为异常来处理了。这些异常继承自Error 类， 此类实现了 Throwable 接口

    • 优化了Zval

      PHP5.5 原来的执行流程是将PHP代码通过词法和语法分析编译成 opcode 字节码， 然后Zend引擎读取这些opcode 指令， 逐条解析执行， 他们在opcode 环节后又引入了类型推断(TypeInf), 然后通过JIT生成 ByteCodes 再执行

      PHP5里面Zval在内存中占据24个字节， 而在PHP7 中优化后的Zval只占16个字节， 这样变量的存储变得非常简单和高效

    • 优化了数组

      PHP5的数组存储形式是一个支持双向链表的HashTable, 不仅支持通过数组的key来做 hash 来做映射访问元素，也能通过foreach 以访问双向链表的方式遍历数据元素

      当我们通过 key 值访问一个元素内容的时候， 有时需要3次指针跳跃才能找到都需要的内容。最重要的一点是， 这些数组元素的存储是分散在各个不同的内存区域的， 因为它们很可能不在同一级缓存中， 导致CPU 不得不到下级甚至内存区域查找，从而引起CPU缓存命中下降

      优化后的 Zend Array 最大的特点是整块的数组元素和 hash 映射表全部连接在一起，被分配在同一内存中

    • 改进了函数调用

      通过优化参数传递的环节减少了一些指令， 提高了执行效率

    3. php-fpm 交互原理

    php-fpm 需要从 cgi 协议开始说起， cgi 协议是是为了解决不同的语言解释器跟 webserver 的通信问题，比如 php-cgi,但是这样 webserver 每一次收到请求， 就会 fork 一个 cgi 进程， 请求结束之后再 kill 这个进程, 这个就特么浪费性能，后面就衍生出了 fast-cgi 协议， fast-cgi 协议就是每次请求处理完之后， 不会kill 这个进程， 而是保留这个进程， 是这个进程可以处理多个请求，这样就减少了重新 fork 进程的操作，

    php-fpm 是 fastcgi 的实现， 并提供了进程管理的功能， 进程包含 master 进程 和 worker 进程。master 进程只有一个， 负责监听端口， 接受来自 webserver 的请求， 而 worker 进程一般会有多个， 每个进程内部都嵌入了一个PHP解释器，是PHP代码真正执行的地方

    在Linux 上， nginx 与 php-fpm 的通信有 tcp socket 和 unix socket 两种方式， tcp socket 的优点是可以跨服务器， unix socket 用于实现同一主机上的进程之间的通信。

    由于 Unix socket 不需要经过网络协议栈， 不需要打包拆包， 计算校验和， 维护序列号和应答等， 只是将应用层数据从一个进程拷贝到另一个进程， 所以效率要比 tcp socket 高， 不过 unix socket 高并发的时不稳定， 连接数爆发时候， 会产生大量的长时缓存， 在没有面向连接协议的支撑下， 大数据包可能直接出错不返回异常

    如果是单机环境， 并且并发量不高(不超过1000), 选择 unix socket 。 如果是高并发， 则选择 tcp socket ， 进行负载均衡， 内核优化等操作

    如果并发较高且想用 unix socket 时， 可以通过下面三个操作提高 unix socket 的稳定性

    • 将 sock 文件挡在 /dev/shm 目录下， 此目录下降 sock 文件放在内存里面， 读写更快
    • 提高 backlog
    • 增加多个sock 文件， 和 php-fpm

    4. php-fpm 如果 杀死了 master 进程， 新的请求会怎么样， 如果杀死了 worker 进程， 新的请求会怎么样？

    如果杀死了 master 进程， 新的请求就不会处理， nginx 会报 504 错误， 后续会报502 (待验证 …)

    如果杀死了 worker 进程， worker 进程不够用的情况下， master 进程会 fork 更多的进程

    5. nginx 504 502 区别?

    502：上游服务器故障, 比如停机, 进程被杀死, 上游服务 reset 了连接， 进程僵死等各种原因

    504：上游服务的执行时间超过了 nginx 的等待时间

    8. 接口和抽象类的区别

    • 接口是对动作的抽象， 而抽象类是对根源的抽象
    • 抽象类要被子类继承， 接口要被子类实现
    • 接口里面只能对方法进行声明， 抽象类既可以对方法进行声明， 也可以对方法进行实现
    • 接口是设计目的是对行为进行约束， 抽象类的目的是代码复用
    • 一个类只能继承一个直接父类， 这个父类可以是具体的类也可以是抽象的类， 但是一个类可以实现多个接口

    9. trait 是什么 以及优先级

    PHP5.4 起， PHP实现了一种代码复用机制，称为trait , trauit 是为了减少单继承语言的限制， 是开发人员可以在不同层次结构内独立的类中复用method

    从基类继承的成员会被trait 插入的成员锁覆盖，优先顺序是来自当前的类覆盖了 trait 的方法， 而 trait 则覆盖了被继承的方法

    11. Laravel 常见的设计模式和原理

    门面模式（Facades）

    在Laravel 应用中， Facade 就是一个可以从容器访问对象的类， 其中核心的部件就是 Facade 类。

    Facade 基类使用了 __callStatic() 魔术方法， 知道对象从容器中被解析出来后， 才会进行调用

    当用户调用 Facade 中的任何静态方法时， Laravel 会从 服务容器 中解析 cache 绑定以及该对象运行所请求的方法

    装饰者模式

    装饰者模式是在开放-关闭原则下实现动态添加或减少功能的一种方式

    Laravel 框架里， 在解析请求生成响应之前或之后需要经过中间件的处理， 主要包括验证维护模式， Cookie加密， 开启会话，CSRF保护等，

    Laravel 中通过 服务容器进行自动实例化

    12. 再聊聊你掌握的设计模式？ 装饰者模式跟适配器的区别

    适配器模式, 状态模式, 工厂模式

    适配器模式

    定义： 将一个类的接口转换成客户希望的另一个接口

    主要角色： 目标接口， 适配者类， 适配器类

    状态模式

    对有状态的对象， 把负责的“判断逻辑” 提取到不同的状态对象中， 允许状态对象在其内部状态发生改变时改变其行为

    装饰者模式跟适配器模式的区别在于， 装饰的目的是增强原本类的功能， 而适配器是对已有对象的接口进行修改， 此外， 装饰还支持递归组合， 适配器则无法实现

    15. InnoDB 跟 MyISAM 的区别？

    InnoDB 支持事务， MyISAM不支持

    16. mysql 的隔离级别， 什么是脏读， 幻读。 怎么样解决脏读跟幻读？

    MySQL 四种隔离级别， 读未提交， 读提交， 可重复读， 串行化

    脏读就是 一个事务读取到了另一个事务未提交的数据， 造成 select 前后数据不一致

    幻读指的是一个事务在前后两次查询同一个范围的时候，后一次查询看到了前一次查询没有看到的行。

    产生幻读的原因是， 行锁只能锁住行， 但是新插入记录的这个动作， 更新的是记录之间的 “间隙” ， 所以解决幻读的办法就是 间隙锁

    innodb_locks_unsafe_for_binlog = 0 # 0 表示开启， 1表示关闭

    17. 索引分类，优缺点， 以及为什么要用索引， 哪些情况下用索引， 索引会在哪些情况下失效

    索引有五种, 主键索引, 普通索引， 唯一索引， 全文索引， 聚合索引

    索引是一种数据结构， 有优点是在一般情况下能更快找出数据， 缺点就是如果维护太多索引， 会影响添加， 更新，删除时候的性能

    一般在经常需要搜索的列， 会经常排序的字段下建索引。

    索引失效的情况

    • 不满足最左原则的时候
    • MySQL 优化器 使用错了索引
    • 搜索列使用了函数
    • 在索引列做了运算操作

    24. redis 缓存雪崩， 穿透， 击穿以及要怎么解决?

    缓存雪崩， 就是所有的key 集中失效

    解决办法就是在每个Key上面加上一个随机值， 或者设置永不过期

    缓存穿透， 就是指缓存和数据库都没有的数据，

    解决办法就是在接口层对不合法的参数做处理， 拦截掉这种情况

    缓存击穿 是指一个Key 非常热点， 当这个Key 失效的瞬间， 持续的大并发就穿破缓存， 直接请求数据库

    解决方案就是 设置热点数据永远不过期， 或者加上互斥锁

    25. redis 阻塞队列

    blpop 是 lpop 的阻塞版本. 如果给定key 内至少有一个非空列表， 那么弹出遇到的第一个非空列表的头元素， 并和被弹出元素所属的列表的名字key 一起， 组成结果返回给调用者

    brpop 是 rpop 的阻塞版本， 该命令会按照给出的Key 顺序查看list , 并在找到第一个非空 list 的尾部弹出一个元素

    blpop 和 brpop 基本是完全一样的, 除了它们一个是从头部弹出元素, 而另一个是从尾部弹出元素

    使用 rpoplpush 命令 ， 或者它的阻塞版本(brpoppush) 来解决队列的不安全问题， 因为它不仅返回一个消息， 同时还将这个消息添加到另一个备份列表当中， 如果一切正常的话， 当一个客户端完成某个消息的处理之后， 可以使用 lrem 命令将这个消息从备份表中删除

    26. redis 持久化

    aof

    aof 是将执行过的写指令记录下来， 在数据恢复的时， 按照从前到后的顺序再讲指令都执行一遍

    通过 redis.conf 里面的 appendonly yes 就可以打开 aof 功能

    默认的aof 持久化策略是 每秒钟 fsync 一次

    因为 aof 采用追加的方式， 所以在不做任何处理的话， aof 文件就会变得越来越大， redis 提供了 aof 重写机制

    rdb

    rdb 是将某一时候的数据持久化到磁盘中， 是一种快照式的持久化方法

    命令是 save , bgsave , redis.conf 里面就是 save 60 1000 表示就是 60s 内至少达到 10000条写命令
